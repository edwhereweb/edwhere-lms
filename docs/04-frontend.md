# 04 — Frontend

This document covers the UI side: the App Router layout, route groups, role-aware navigation, the component library, and conventions for building pages and forms.

## 1. Tech stack quick reference

- **Next.js 14 App Router** with React Server Components.
- **TypeScript strict.**
- **Tailwind CSS** for styling. `tailwind.config.ts` extends the standard preset with theme tokens used by shadcn/ui.
- **shadcn/ui** components live in `components/ui/`. **Do not edit them by hand** — they are generated by the shadcn CLI. Compose, don't fork.
- **lucide-react** for icons.
- **React Hook Form + Zod** for forms.
- **Zustand** for the small amount of client-side state we have (e.g. `hooks/use-confetti-store.ts`).
- **React Context** for theming (`components/providers/theme-provider.tsx`).
- **react-hot-toast** for transient notifications.
- **react-confetti** triggered via `confetti-provider.tsx` for course-completion celebration.
- **Editors:** React Quill (legacy), Tiptap (`@tiptap/*`) for rich text, Monaco (`@monaco-editor/react`) for HTML embed editing.
- **Players:** `@mux/mux-player-react` for Mux video; YouTube via raw iframe; PDFs via `components/pdf-viewer.tsx`.

## 2. Top-level layout

`app/layout.tsx` is the root layout that wraps the entire app:

- `ClerkProvider` (auth context).
- `ThemeContextProvider` (light/dark mode).
- `ConfettiProvider` (global confetti portal).
- `ToastProvider` (`react-hot-toast` mounting point).
- Global metadata: title, description, OpenGraph, Twitter, robots, icons.
- Global CSS + fonts.

Below the root, four **route groups** partition the app. Route groups are folders wrapped in `()` that don't appear in the URL, so they're used purely to share a layout and to organise files by audience.

```
app/
├── layout.tsx                    Root: providers, metadata, fonts
├── icon.png                      Browser tab icon
├── api/                          REST handlers (see 05-backend-api.md)
│
├── (auth)/                       Sign-in / sign-up
│   ├── layout.tsx                Centered Clerk component wrapper
│   └── (routes)/
│       ├── sign-in/[[...sign-in]]/page.tsx
│       └── sign-up/[[...sign-up]]/page.tsx
│
├── (public)/                     Marketing surface (no login required)
│   ├── layout.tsx                PublicNavbar + footer
│   ├── courses/                  /courses, /courses/[id]
│   └── blog/                     /blog, /blog/[slug]
│
├── (dashboard)/                  Authenticated app (student + staff)
│   ├── layout.tsx                Top Navbar + left Sidebar (role-aware)
│   ├── _components/              Navbar, Sidebar, SidebarRoutes, MobileSidebar
│   └── (routes)/
│       ├── (root)/page.tsx       Logged-in landing
│       ├── dashboard/            Student dashboard
│       ├── search/               Course search/browse for logged-in users
│       ├── contact/              Internal contact form
│       ├── teacher/              Teacher + admin pages
│       ├── marketer/             Marketer pages
│       └── blogger/              Blogger pages
│
└── (course)/                     Course consumption "player" experience
    └── courses/[courseId]/
        ├── layout.tsx            CourseNavbar + CourseSidebar
        ├── start/page.tsx        Redirect to first published chapter
        ├── chapters/[chapterId]/page.tsx
        └── chat/page.tsx         Mentor Connect chat
```

The convention `(...)/(routes)/...` you see inside `(dashboard)` is just an extra layer of grouping for organisation; nothing magical.

## 3. Layouts and what they do

| Layout                                         | Wraps                  | Responsibilities                                                                                    |
| ---------------------------------------------- | ---------------------- | --------------------------------------------------------------------------------------------------- |
| `app/layout.tsx`                               | Everything             | Providers, fonts, global metadata.                                                                  |
| `app/(auth)/layout.tsx`                        | `/sign-in`, `/sign-up` | Centers Clerk auth components.                                                                      |
| `app/(public)/layout.tsx`                      | `/courses`, `/blog`    | `PublicNavbar` + footer. No auth required.                                                          |
| `app/(dashboard)/layout.tsx`                   | All authed dashboards  | Fetches `safeProfile`. Renders fixed `Navbar` + fixed `Sidebar`. Hidden if user has no profile yet. |
| `app/(dashboard)/(routes)/teacher/layout.tsx`  | All `/teacher/*`       | Hard-redirects non-TEACHER/non-ADMIN users to `/`. Single role gate for the whole subtree.          |
| `app/(dashboard)/(routes)/marketer/layout.tsx` | All `/marketer/*`      | Hard-redirects non-MARKETER/non-ADMIN users.                                                        |
| `app/(dashboard)/(routes)/blogger/layout.tsx`  | All `/blogger/*`       | Hard-redirects users without `canManageBlogs`.                                                      |
| `app/(course)/courses/[courseId]/layout.tsx`   | The player             | Fetches course + modules + chapters + user progress + unread message counts. Renders nav + sidebar. |

Layouts are Server Components and run on every navigation within their subtree. They are the right place for role-gating and data hydration that all child pages need.

## 4. Role-aware navigation

The navbar is single source of truth for "what mode am I in" / "what mode can I switch to". See `components/navbar-routes.tsx`:

- It uses `usePathname()` to detect whether we're under `/teacher`, `/marketer`, `/blogger`, or `/courses/[id]/...` (player).
- It calls a server endpoint (or reads a passed-in profile) to know the current user's role.
- It conditionally renders **Mode** buttons (e.g. "Teacher Mode", "Blogger Mode") for users with the matching role, plus an **Exit** button when inside a mode.
- Always renders Clerk's `<UserButton/>` in the corner.

The `Sidebar` (`app/(dashboard)/_components/sidebar.tsx`) uses `SidebarRoutes` to render different link sets depending on the URL prefix and the user's role. This is the simplest way to read what's available to whom.

### 5.2 Auth

- `/sign-in/...`, `/sign-up/...` → Clerk-rendered.

### 5.3 Student dashboard

- `/dashboard` → enrolled courses split into "In Progress" / "Completed".

### 5.4 Course consumption (any logged-in user with access)

- `/courses/[courseId]/start` → redirects to the first published chapter.
- `/courses/[courseId]/chapters/[chapterId]` → the player. Branches on `chapter.contentType` to render:
  - `VIDEO_MUX` → `VideoPlayer` (Mux Player).
  - `VIDEO_YOUTUBE` → embedded YouTube iframe.

  - `HTML_EMBED` → `HtmlEmbedPreview`.
  - `PDF_DOCUMENT` → `PdfViewer`.

  - `EVALUATION` → `QuizPlayer`.

Under `/teacher`, gated by the teacher layout. Highlights:

## 6. Server Components vs Client Components

Default: everything is a **Server Component**. A component becomes a Client Component only when it has `"use client"` at the top. Use the client only when you need:

- React state or effects.
- Browser APIs (window, fetch from the browser, dropzone, file inputs, players).
- Event handlers attached to DOM (forms, buttons triggering mutations).

Patterns you'll see repeatedly:

- **Page (Server)** fetches data via `actions/*` and passes it as props to a colocated **`_components/foo-form.tsx` (Client)** which handles edit UI.
- **List page (Server)** fetches rows and renders a **Client table component** (e.g. `data-table.tsx`) for sorting/filtering interactions.
- **Player chapter page (Server)** branches on content type and dynamically renders a per-type **Client** sub-component.

## 7. Data fetching boundaries

| Where you are                | What to use                                                                           |
| ---------------------------- | ------------------------------------------------------------------------------------- |
| Server Component             | Direct call to `actions/*` (which use `lib/db`) or `db` directly.                     |
| Client Component (mutation)  | `axios` / `fetch` to `/api/...`, then `router.refresh()` to re-render server data.    |
| Client Component (read-only) | Prefer prop-passing from server. Only fetch from client when interaction requires it. |

Avoid mixing: don't import server-only code (anything that uses `db`) inside a client component.

## 8. Forms

The standard form pattern across the codebase:

1. Define the Zod schema in `lib/validations.ts` (server-side validation source of truth).
2. In the client form component, use `useForm({ resolver: zodResolver(schema) })`.
3. Use `<Form>`, `<FormField>`, etc. from `components/ui/form.tsx` (shadcn) for layout + field error rendering.
4. On submit, `await axios.patch('/api/...', values)` (or POST/PUT/DELETE).
5. On success: `toast.success('Saved')`, `router.refresh()`. Sometimes also `setIsEditing(false)` for inline-edit forms.
6. On error: `toast.error('Something went wrong')` per `AGENTS.md`. Do **not** swallow with `.catch(() => {})`.

Inline edit forms (e.g. `chapter-description-form.tsx`, `course-image-form.tsx`) are the dominant pattern for editing course/chapter fields. Each field gets its own micro-form for atomic saves.

## 10. Styling conventions

- Tailwind-only. No CSS modules, no styled-components.
- Use the `cn(...)` helper from `lib/utils.ts` for conditional classes.
- Theme colors come from CSS variables defined in `app/globals.css` and consumed by Tailwind's `theme.extend.colors` (shadcn pattern). Use semantic classes like `bg-background`, `text-foreground`, `bg-card`, etc., not raw hex.
- Dark mode is class-based (`<html class="dark">`); toggled by `theme-switch.tsx` via `next-themes`.
- For "fixed top + side + scroll body" layouts, follow `(dashboard)/layout.tsx` and `(course)/courses/[id]/layout.tsx` — they pin the navbar and sidebar with `fixed`/`inset-y-0` and add padding to the main container.

## 11. SEO and metadata

- The root `metadata` is set in `app/layout.tsx` (title, description, OG, robots, icons).
- Per-page overrides: export an `async generateMetadata({ params })` from a Server Component. The public course detail page (`app/(public)/courses/[courseId]/page.tsx`) is the reference example — it also injects JSON-LD via a `<script type="application/ld+json">` tag for structured data.
- Public course slugs (`Course.slug`) drive friendly URLs. The detail page accepts both slug and ObjectId in the param.

## 12. Image and asset rules

- `next.config.js` whitelists `utfs.io` (legacy UploadThing) and `img.clerk.com` for `next/image`. Anything else won't be optimised.
- R2 files are served from same-origin `/api/files/<key>`. They work with `<Image src=... unoptimized />` or via plain `<img>`. Don't add R2 to `remotePatterns` — the proxy route handles them.
- `public/` holds static assets shipped with the build (`edwhere-logo.png`, etc.). Reference as `/edwhere-logo.png`.

## 13. Conventions checklist for new components

When adding a new component:

- File name: `kebab-case.tsx`. Named export `PascalCase`.
- If it's page-specific, put it in `_components/` next to the page (the `_` keeps it out of the route table).
- If it's reusable across routes, put it in `components/`.
- Default to a Server Component. Add `"use client"` only when needed.
- For icons, prefer `lucide-react` (no SVG sprawl).
- For new data displays involving forms, use `react-hook-form + zod` + `components/ui/form.tsx`.
- For new mutations from the client, follow the toast + `router.refresh()` pattern.
- Don't import from `components/ui` to extend its styling — wrap it in your own component instead.

## 14. Things to be careful about

- **Layout caching:** dashboard layout uses `dynamic = 'force-dynamic'` to prevent stale auth state. If you add a new authenticated layout, set the same.
- **`router.refresh()` after mutation:** without it, Server Components keep the stale snapshot and the user thinks the change didn't happen. Always call it after PUT/POST/PATCH/DELETE.
- **Order with delete + navigate:** if a mutation deletes the current page's data, `router.push(elsewhere)` BEFORE `router.refresh()`. Otherwise the refresh tries to render a deleted resource.
- **Loading states:** the codebase relies heavily on `useState(isLoading)` in client components. There's no global loading provider. Keep buttons `disabled` while loading and surface the state with a spinner or skeleton.
- **Error UX:** every catch should at minimum show a `toast.error('Something went wrong')`. Per `AGENTS.md`, never swallow with `.catch(() => {})`.
- **Hydration mismatches:** dates/times displayed in dashboards should be formatted server-side (use `date-fns`) or wrapped in a client-only component to avoid timezone drift.
